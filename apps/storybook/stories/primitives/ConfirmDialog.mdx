Locked destructive / irreversible confirmation pattern. Composed from
`Dialog` primitives (sm size, title, optional description, Cancel + Confirm).
This is the pattern Errors.md Rule 2 prescribes for "user must make a
decision before continuing" — use it for every destructive flow in the
product so the system reads identically wherever it appears.

## When to use

Per Errors.md, ConfirmDialog is for **decisions** — Dialog tier in the
error-surface hierarchy:

| Need | Use |
| --- | --- |
| Confirmable info ("Saved successfully") | `Toast` |
| Persistent state ("Trial expires soon") | `Banner` |
| Ephemeral feedback | `Toast` |
| **Decision required before continuing** | **`ConfirmDialog` / `Dialog`** |

ConfirmDialog handles the common case (one-question confirmation). For
multi-field decisions, drop down to `Dialog` directly.

## Anatomy

```tsx
import { ConfirmDialog, Button } from "@azeer/ui";

<ConfirmDialog
  title="Delete conversation?"          // sentence case + ? per Casing.md
  description="Sara Khan will be removed from your contact list. This cannot be undone."
  confirmLabel="Delete"                  // verb, never "OK" / "Submit"
  cancelLabel="Cancel"                   // default; "Keep editing" / "Go back" also common
  destructive                            // → confirm button = variant="destructive"
  onConfirm={() => deleteConversation()}
  trigger={<Button variant="destructive">Delete</Button>}
/>
```

## Title + button copy

Per Casing.md:

| Surface | Form |
| --- | --- |
| **Title** | Sentence case, ends with `?` for questions, no period for statements |
| **Description** | Full sentence(s), period at end |
| **Confirm button** | Verb + object (or just verb). "Delete", "Discard", "Send", "Close". **Never "OK" / "Submit" / "Yes"** |
| **Cancel button** | "Cancel" by default; "Keep editing" / "Go back" for discard flows |

Examples:

- "Delete conversation?" → Confirm: "Delete"
- "Discard unsaved changes?" → Confirm: "Discard", Cancel: "Keep editing"
- "Send broadcast?" → Confirm: "Send" (primary, not destructive)
- "Close conversation?" → Confirm: "Close"

## Controlled vs. uncontrolled

```tsx
// Uncontrolled (with trigger)
<ConfirmDialog
  trigger={<Button variant="destructive">Delete</Button>}
  title="…"
  onConfirm={…}
/>

// Controlled (programmatic — e.g. dirty-state guard before navigation)
const [open, setOpen] = useState(false);
<ConfirmDialog
  open={open}
  onOpenChange={setOpen}
  title="…"
  onConfirm={() => { … ; setOpen(false); }}
/>
```

## A11y

- Inherits all Dialog plumbing (focus trap, Esc to close, outside-click to dismiss, aria-labelledby/describedby).
- Primary CTA (Confirm) does NOT auto-focus on open — the user explicitly clicks Confirm to acknowledge. This avoids accidental confirmations from Enter-press.
- For irreversible destructive actions, consider doubling the safeguard with a "type the conversation name to confirm" input — beyond ConfirmDialog's scope; build with raw `Dialog` when needed.

## Forbidden

- ConfirmDialog with non-destructive language ("Are you sure?") — be specific about the action ("Delete?")
- "OK" / "Yes" / "Submit" as confirm labels — use the verb
- ConfirmDialog with >1 question — break into a multi-step Dialog
- ConfirmDialog where the action is reversible (e.g. "Archive") — those don't need confirmation; show an undo Toast instead
- Toasting a destructive action — Toast auto-dismisses; users miss it. Always use ConfirmDialog for irreversible decisions.
- Disabling the Confirm button while it processes — pass `loading` to a custom `Dialog` instead, or split into multiple states
