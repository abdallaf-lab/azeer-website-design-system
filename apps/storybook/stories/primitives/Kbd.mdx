left-to-right universally, so `Kbd` forces `dir="ltr"` on its element. Use

## Modifier conventions

| Key | Render as |
| --- | --- |
| Command (macOS) | `⌘` (exact glyph) |
| Control | `Ctrl` |
| Shift | `Shift` |
| Alt / Option | `Alt` |
| Escape | `Esc` |
| Enter | `Enter` |
| Tab | `Tab` |
| Arrow up / down / left / right | `↑` / `↓` / `←` / `→` (exact glyphs) |

Combine with `+`: `⌘K`, `Ctrl+Shift+P`, `Cmd+,`. Sentence-case the words,
exact glyphs for symbols. Lowercase modifier names (`cmd`, `ctrl`, `esc`)
are forbidden.

| Token | Where |
| --- | --- |
| `--font-mono` (JetBrains Mono) | Monospace shortcut text |

## A11y

- Renders a real `<kbd>` element — the correct semantic for keyboard input.
- `dir="ltr"` set on the element so the modifier sequence renders the same inside Arabic prose. See the **Inside Arabic prose** story above — flip the toolbar to RTL and the `⌘K` token stays LTR while the surrounding Arabic text mirrors.
- `select-none` prevents the modifier glyph from being included in a text selection drag through prose.

## Forbidden

- Lowercase modifier names (`cmd`, `ctrl`, `shift`, `esc`) — exact case per the table
- Spelling out symbols ("Command + K") — use `⌘K`
- Mixing the `+` separator with spaces (`⌘ K`, `⌘ + K`) — `⌘K` or `Ctrl+Shift+P`
- Wrapping a directional context around `Kbd` — it always renders LTR
- Custom `<span>` with kbd-like styling — use the `Kbd` primitive
