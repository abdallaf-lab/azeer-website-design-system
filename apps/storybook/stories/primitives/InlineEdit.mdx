Click-to-edit pattern. Display mode shows the value as text with a hover-
revealed `Pencil` affordance; clicking enters edit mode (Input + Save /
Cancel). `Enter` commits, `Esc` cancels.

Used for settings titles, contact names, conversation subjects — anywhere
the static value is the resting state and editing is a deliberate
interaction.

## API

```tsx
const [name, setName] = useState("Sara Khan");

<InlineEdit
  value={name}
  onValueChange={setName}
  validate={(v) => v.trim() === "" ? "Name can't be empty." : null}
  placeholder="Click to add a name"
/>
```

| Prop | Type | Notes |
| --- | --- | --- |
| `value` | `string` | Current value |
| `onValueChange` | `(value) => void` | Commits on Save click / Enter — only fires when `validate` returns null |
| `validate` | `(value) => string \| null` | Return an error string to block save + show inline; return `null` to commit |
| `placeholder` | `string` | Display-mode placeholder shown italic + `fg-subtle` when value is empty. Default `"Empty"` |
| `inputPlaceholder` | `string` | Edit-mode Input placeholder. Defaults to `placeholder` |
| `size` | `"sm" \| "md" \| "lg"` | Passed to the Input — pick to match surrounding typography |
| `disabled` | `boolean` | Disables the click-to-edit trigger |
| `ariaLabel` | `string` | `aria-label` for the trigger when the visible text isn't enough on its own (e.g. a contact name without context) |
| `textClassName` | `string` | Class names for the static display element (e.g. `"text-heading-md"`) |

## Interaction model

| State | Behavior |
| --- | --- |
| **Display (resting)** | Text shown plain; `Pencil` icon hidden |
| **Display (hover / keyboard-focus)** | Background → `state-hover` overlay; `Pencil` icon fades in |
| **Click / Enter on trigger** | Swap to Input + Save + Cancel; Input auto-focuses |
| **Type + Enter** | Run `validate`; commit on success / show error inline |
| **Type + Esc** | Revert draft + return to display mode |
| **Click Save** | Same as Enter |
| **Click Cancel** | Same as Esc |
| **Click outside while editing** | (Phase 4) no special behavior — Save / Cancel are explicit. Consumer can wire if needed via focus-loss detection. |

## Tokens used

Inherits from `Input` and `Button` `sm` / `secondary`. Adds:

| Token | Where |
| --- | --- |
| `--color-state-hover` | Display-mode hover overlay |
| `--color-fg-default` | Static text color |
| `--color-fg-subtle` | Empty-placeholder text (italic) |
| `--color-fg-muted` | Pencil icon (hover-revealed) |
| `--color-danger-text` | Validation error message |
| `--text-body-xs` | Validation error typography |

## A11y

- Display trigger is a real `<button>` — keyboard focus shows the Pencil affordance + global focus ring.
- Edit mode's Input auto-focuses on mount (`autoFocus`).
- `Enter` / `Esc` shortcut handlers stop propagation so the keystrokes don't escape to parent forms.
- `aria-invalid` set on the Input when `validate` returns an error.
- For long text, the trigger inherits the parent's truncation behavior — pass `truncate` via `textClassName` if needed.

## Forbidden

- InlineEdit for fields that should always be visible inputs (form fields) — use `Input` inside `FormField` instead
- InlineEdit nested inside a clickable row (entire row triggers something else) — clicks conflict
- InlineEdit for password-style or sensitive values — use a `Dialog` with explicit Cancel / Save
- Inline validation on every keystroke — `validate` runs on Save attempt; live validation is hostile UX per Forms.md §4
- Wrapping the trigger in a heading element directly (e.g. `<h2><InlineEdit /></h2>`) — InlineEdit's display element IS the heading-tier text; the trigger is a button. Style via `textClassName` instead.
