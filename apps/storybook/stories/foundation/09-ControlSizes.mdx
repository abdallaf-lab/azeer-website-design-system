# Control Sizes

**3 sizes — `sm` 32px / `md` 40px / `lg` 48px.** Not 2. Not 4. Three is the
minimum that covers dense product UI + form-level controls + hero CTAs without
forcing engineers to invent off-scale heights mid-flight.

<ControlSizesScale />

## Rules

1. **`md` is the default. Always.** Render `md` if no `size` prop is passed — never write `size="md"` explicitly.
2. **`sm` for dense surfaces** — when the control lives inside another control's rhythm (toolbar, table-row, inline filter row, popover with multiple buttons). Rule of thumb: if the parent row is 40px or shorter, use `sm` controls inside it.
3. **`lg` only on hero CTAs** — auth pages, onboarding, mobile-first primary actions. **Never** in form modal action rows, page-header CTAs, or dashboard cards.
4. **Mixed sizes in the same row are forbidden** — if a row has multiple controls, they must all be the same size. Mixed sizes break visual alignment.
5. **Icon size scales with control** — `sm` → 14px, `md` → 16px, `lg` → 20px. The wrapper handles it; don't override.

<Callout intent="warning" title="Toggles live outside this scale">
Checkbox (16×16), Radio (16×16), and Switch (36×20) have one canonical size
each — they don't take a `size` prop. They sit inside form rows that scale
with the parent FormField.
</Callout>

## Density mode (future hook — v1.2+)

```css
[data-density="compact"] {
  --control-h-sm: 28px;   /* was 32 */
  --control-h-md: 32px;   /* was 40 — compact treats md as a new sm */
  --control-h-lg: 40px;   /* was 48 */
}
```

The token structure supports compact mode without component rewrites. Don't
ship it before v1.2 — the scaffold attribute is in place but the overrides
intentionally don't.

## Forbidden

- `size="xs"` / `size="xl"` — only `sm` / `md` / `lg` (+ `icon-sm` / `icon-md` / `icon-lg`)
- Passing `size="md"` explicitly (it's the default — passing it is noise)
- Inline height overrides on controls: `className="h-9"` on a Button
- Mixing sizes in the same row
- A `size` prop on Checkbox / Radio / Switch / Slider
